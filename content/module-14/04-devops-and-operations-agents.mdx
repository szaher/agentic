## Overview

In Lesson 03, you explored knowledge and support agents that help users find information and resolve issues. This lesson covers DevOps and software operations agents—where agent autonomy is most consequential because a wrong action can take down production systems.

Module 13 introduced computer use agents and coding agents for general automation. This lesson goes deeper into enterprise DevOps, where agents must balance operational efficiency with safety controls. You'll learn how incident response, code review, CI/CD, and runbook automation agents operate within safety tiers that match the blast radius of their actions.

> DevOps agents operate in production environments where errors have real consequences: downtime, data loss, security breaches, and revenue impact.

## Incident Response Agents

Incident response agents automate the initial triage and remediation steps when alerts fire. Their goal is to reduce mean time to resolution (MTTR) by handling routine incidents autonomously and escalating complex issues to human operators with full context.

### The Incident Response Flow

When an alert fires, the incident response agent follows this sequence:

1. **Alert Classification** — Determine severity (P1 critical outage, P2 major degradation, P3 minor issue, P4 informational) based on service impact, affected users, and business hours.

2. **Context Gathering** — Collect logs from affected services, metrics (CPU, memory, request latency, error rates), recent deployment history, and known issues.

3. **Runbook Matching** — Match the alert type and context to known remediation procedures. For example, "high memory usage on service X" maps to "check for memory leaks, restart if OOM, scale replicas if sustained load."

4. **Remediation Execution** — Execute runbook steps within the agent's safety tier. Read-only actions (check logs, list resources) proceed automatically. Reversible actions (scale replicas, toggle feature flags) proceed with logging. Irreversible actions (delete resources, rollback deployments) require human approval.

5. **Escalation** — If remediation fails or the issue is unrecognized, escalate to the on-call engineer with all gathered context.

### Alert Triage and Classification

Alert classification determines how the agent proceeds. High-severity alerts (P1, P2) trigger immediate action and human notification. Lower-severity alerts (P3, P4) may be queued for business hours or handled entirely by the agent.

Classification criteria include:

- **Service Impact** — Is a customer-facing service down? Are internal tools affected?
- **Affected Users** — How many users are impacted? Are they external customers or internal employees?
- **Business Hours** — Is this during peak traffic or off-hours?
- **Known Issues** — Does this match a recent deployment or a known bug?

## Code Review Agents

Code review agents augment human reviewers by automating mechanical checks and surfacing issues that require domain expertise. They integrate with version control systems to analyze pull requests before they merge.

### Beyond Simple Linters

While linters check syntax and style, enterprise code review agents understand:

- **Business Context** — "This API endpoint handles payment processing, so it must include fraud detection checks per our security policy."
- **Internal API Conventions** — "This service uses our internal RPC framework, which requires request IDs for tracing. This call is missing one."
- **Organization-Specific Security Policies** — "This function logs user data, which violates our PII retention policy. Use the sanitized logging wrapper instead."

Code review agents also perform static analysis (null pointer risks, resource leaks), security scanning (dependency vulnerabilities, hardcoded secrets), and style enforcement (naming conventions, documentation requirements).

### Integration with Human Review

Code review agents do not replace human reviewers. They handle mechanical checks so humans can focus on design decisions, algorithmic correctness, and maintainability. The typical flow:

1. Developer opens pull request.
2. Agent runs automated checks (tests, linters, security scans, policy validation).
3. Agent posts inline comments on specific lines that violate policies or conventions.
4. Developer addresses agent feedback.
5. Human reviewer evaluates design, logic, and context that agents cannot assess.

## CI/CD Agents

Continuous Integration and Continuous Deployment (CI/CD) agents automate build, test, and deployment workflows. They triage failures, detect flaky tests, and manage deployment gates.

### Build Failure Triage

When a build fails, the CI/CD agent:

1. **Identifies the Failing Test** — Parse build logs to locate the test name, failure message, and stack trace.
2. **Analyzes the Failure Pattern** — Is this a new failure or a known flaky test? Did it fail on one platform but pass on others?
3. **Searches for Related Code** — Locate the test file, the code under test, and recent changes that might have caused the failure.
4. **Suggests Fixes** — Propose a fix based on the failure type (assertion mismatch, timeout, dependency issue).

This reduces developer context-switching: instead of manually digging through logs, the developer receives a summary with suggested next steps.

### Flaky Test Detection

Flaky tests pass or fail non-deterministically, often due to race conditions, timing dependencies, or environmental state. CI/CD agents detect flaky tests by tracking test results across multiple runs:

- A test that fails 30% of the time is flaky.
- A test that fails consistently after a specific commit is a real regression.

Once detected, flaky tests are flagged, quarantined (run separately to avoid blocking CI), or automatically retried.

### Deployment Gate Management

Before deploying to production, CI/CD agents check deployment gates:

- **Test Coverage** — Does the new code meet the required coverage threshold (e.g., 80% line coverage)?
- **Performance Benchmarks** — Do response times stay within acceptable bounds?
- **Security Scans** — Are there known vulnerabilities in dependencies?
- **Manual Approval** — For high-risk changes, does a human need to approve?

The agent makes automated go/no-go decisions for routine deployments and escalates risky deployments to human operators.

## Runbook Automation with Safety Tiers

Runbooks are documented procedures for handling operational tasks: restarting a service, scaling infrastructure, rolling back a deployment. DevOps agents automate runbook execution, but not all runbook actions are equally safe.

### Three Safety Tiers

DevOps actions fall into three tiers based on reversibility and blast radius:

1. **Read-Only** — Querying logs, checking metrics, listing pods, inspecting configurations. These actions observe state without modifying it. **Auto-execute.**

2. **Reversible** — Scaling replicas (can scale back down), toggling feature flags (can toggle back), restarting services (service recovers). These actions modify state but can be undone. **Execute with logging.**

3. **Irreversible** — Deleting resources (cannot restore deleted data), database schema changes (migrations are hard to reverse), rollback deployments (can cause data inconsistency). These actions cannot be undone or have severe consequences. **Require human approval.**

This is the blast radius pattern from Module 12 (Production, Deployment and Safety): match autonomy to the reversibility of the action.

## Enterprise Controls Used in This Vertical

DevOps agents operate in high-stakes environments where errors can cause outages, data loss, or security breaches. Enterprise controls ensure agents operate safely:

- **Blast-Radius Limits** — Actions are categorized by safety tier (read-only, reversible, irreversible). High-risk actions require human approval.
- **Change Management Approval Gates** — Deployments to production, schema changes, and infrastructure modifications go through approval workflows.
- **Incident Audit Trail** — Every action the agent takes is logged with timestamp, user context, and outcome. Logs are immutable and retained for compliance.
- **Rollback Authorization** — Only authorized users (e.g., on-call engineers, release managers) can approve rollbacks or irreversible operations.

These controls mirror the production safety patterns from Module 12: start with restrictive defaults, expand autonomy incrementally, and always maintain an audit trail.

## Summary

DevOps and software operations agents automate incident response, code review, CI/CD workflows, and runbook execution. They reduce toil and improve reliability, but they operate in environments where mistakes have real consequences.

The key to safe DevOps automation is safety tiers: read-only actions proceed automatically, reversible actions execute with logging, and irreversible actions require human approval. This ensures agents stay within safe boundaries while maximizing operational efficiency.
