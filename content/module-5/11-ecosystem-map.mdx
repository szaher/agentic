## Overview

You've built a custom harness from scratch. You traced eight concerns — state, routing, tools, errors, events, interrupts, delegation, and persistence — from first principles through production features. Now you'll see how the industry solves the same problems.

This lesson compares real-world harnesses across the eight-concern framework you learned in Lessons 1–8. You'll categorize frameworks, read a side-by-side comparison table, understand the "Custom" column, track ecosystem evolution, and identify trend lines that matter for production deployments.

<Callout type="info">
**Ecosystem Snapshot**

This comparison reflects the landscape as of mid-2026. Framework capabilities evolve rapidly. Verify against current documentation before production decisions.
</Callout>

## Four Harness Categories

Agent harnesses cluster into four categories, each with different primitives and integration points. Understanding the categories helps you match frameworks to your architecture.

### Agent SDKs / Model-Adjacent Runtimes

**Lower-Level Model APIs** provide direct access to tool use without managed execution:

**Anthropic Messages API** is the foundational protocol for Claude. You send messages with a `tools` array, receive `tool_use` blocks, execute tools in your code, return `tool_result` messages, and repeat until the model returns a text block. State, routing, error recovery, and persistence are your responsibility. This API gives you full control and minimal abstraction.

**Key capabilities:**

- Direct protocol access with no managed runtime overhead
- Full control over message history and conversation state
- Manual tool execution with custom validation and error handling
- Provider-specific optimizations (prompt caching, batching)
- Zero vendor lock-in for execution logic

**Managed Agent SDKs** wrap the protocol loop and add session continuity:

**OpenAI Agents SDK** provides a managed runtime for multi-turn conversations with persistent threads stored server-side. You define tools, configure assistants, and call `run()`. The SDK handles tool execution, streaming, and thread state. Persistence is managed by OpenAI's infrastructure.

**Key capabilities:**

- Streaming support with token-by-token output
- Native tool-call parsing and validation
- Server-side thread persistence with automatic state management
- Built-in retry policies and error recovery
- Managed file storage for tool outputs

**Claude Agent SDK** offers file-based checkpointing and session continuity for local or self-hosted deployments. You define tools, run agents in isolated environments, and resume from checkpoint files. Persistence is file-based by default; full run-state durability requires external storage.

**Key capabilities:**

- File-based checkpointing for session recovery
- Isolated execution environments per agent session
- Resume from checkpoint after crashes or restarts
- Local or self-hosted deployment with no cloud dependencies
- Event hooks for custom observability and logging

**UI/App Agent SDKs** integrate LLM agents into application frontends:

**Vercel AI SDK** provides React hooks (`useChat`, `useCompletion`), streaming UI components, and server-side route handlers for Next.js applications. It abstracts model provider differences (OpenAI, Anthropic, Google, etc.) and simplifies tool integration for web apps. State is typically managed client-side or in framework state.

**Key capabilities:**

- React hooks for streaming chat and completions in UI components
- Provider-agnostic abstractions across OpenAI, Anthropic, Google, and others
- Client-side state management hooks with optimistic updates
- Server-side route handlers for Next.js API endpoints
- Built-in streaming UI components for chat interfaces

### Graph Runtimes

**LangGraph** models agents as state graphs. You define nodes (agent steps or tool calls), edges (control flow), and state schemas. LangGraph provides:

- **Checkpointers:** save/restore state at every step for durability and debugging.
- **Stores:** persistent key-value storage for cross-run data.
- **Interrupts:** pause execution for human approval or external input.
- **LangSmith:** observability and tracing for graph executions.

Graph-based harnesses make state transitions explicit. You see exactly where the agent is in the graph, what state it carries, and where it can branch next. This explicitness supports visual debugging, controlled interrupts, and deterministic retries.

**Key capabilities:**

- Typed state schemas with compile-time validation
- Visual graph inspection for runtime debugging
- Conditional branching based on state predicates
- Time-travel debugging by replaying from checkpoints
- Subgraph composition for modular agent architectures

### Workflow / Orchestration Runtimes

Workflow engines designed for distributed systems can host agent steps:

**Temporal** is a durable execution platform. You write workflows as code (in Python, TypeScript, Go, etc.), and Temporal guarantees exactly-once execution, automatic retries, and recovery from failures. LLM integrations treat model calls as workflow activities. Temporal handles persistence, replay, and scheduling. This is useful when agents are part of larger distributed workflows (e.g., order processing with human-in-the-loop reviews and LLM-generated summaries).

**Prefect / Airflow** are workflow orchestrators for data pipelines. You can embed LLM agent steps into tasks, schedule runs, and observe execution graphs. These frameworks excel at batch processing and scheduled workflows, but don't provide interactive session continuity.

Workflow runtimes complement agent SDKs: use agent SDKs for conversational loops, workflow runtimes for long-running, multi-stage processes with external dependencies.

**Key capabilities:**

- Exactly-once execution semantics for all workflow steps
- Automatic retry with exponential backoff for transient failures
- Compensation logic for rolling back multi-step transactions
- Scheduled triggers for time-based workflow execution
- Cross-service orchestration with durable state guarantees

### Multi-Agent Runtimes

Multi-agent frameworks coordinate multiple agents working together:

**CrewAI** models teams of agents with roles, goals, and delegation. You define a crew (e.g., researcher + writer + reviewer), assign tasks, and let agents delegate subtasks to each other. CrewAI handles task routing and agent collaboration.

**AG2 / AutoGen Lineage** (formerly Microsoft AutoGen, now in maintenance mode with community forks like AG2) supports conversational agents that exchange messages in a group chat. You define agents with system prompts, tools, and conversation policies. AG2 manages message routing and turn-taking.

**OpenAI Agents SDK Handoffs** allow one agent to transfer control to another by returning a handoff response. The SDK routes the conversation to the target agent. This is a lightweight multi-agent pattern for single-provider deployments.

Multi-agent runtimes add orchestration overhead but enable specialization: one agent handles research, another writes reports, another reviews for quality.

**Key capabilities:**

- Role-based task assignment with agent specialization
- Inter-agent message passing with structured protocols
- Delegation chains for hierarchical task decomposition
- Shared context pools for cross-agent state
- Conversation policies for turn-taking and conflict resolution

## Reading the Comparison Table

The comparison table maps how each framework addresses the eight concerns. Understanding how to read the table helps you evaluate frameworks for your requirements.

**Cell Semantics:** Each cell shows the framework's *approach* to a concern, not a quality rating. "Server-side threads" (OpenAI Agents SDK state management) is different from "Manual state in app" (Anthropic Messages API), but neither is inherently better — they reflect different design philosophies and deployment models.

**Managed vs Manual vs None:**

- **Managed:** The framework handles the concern for you. Example: OpenAI Agents SDK manages thread persistence server-side; you don't write checkpoint code.
- **Manual:** You write the implementation. Example: Anthropic Messages API requires manual error handling in `tool_result` messages.
- **None:** The framework doesn't support the concern. Example: OpenAI Agents SDK has no native interrupt primitives.

**The "Developer-Defined" Custom Column:** "Developer-defined" means you make all the decisions — which is both the hardest and most flexible option. You choose the state schema, the checkpoint format, the retry policy, the event format, and the persistence backend. Frameworks give you primitives; the Custom column reminds you that even with primitives, you're still writing custom integration code.

**How to Read the Table:** Read row-by-row (by concern) when evaluating frameworks, not column-by-column. For each concern that matters to your application, compare how frameworks handle it. If you need durable execution, read the Persistence row. If you need human-in-the-loop workflows, read the Interrupts row. If you need multi-agent coordination, read the Delegation row. The table is a decision aid, not a scorecard.

## The "Custom" Column

Notice that "Custom" appears in every row. This is not a limitation of the comparison — it's a fundamental truth about agent harnesses.

**Every harness is custom code.** Even when you use a framework, you're writing custom tool definitions, custom routing logic, custom error handlers, custom state schemas, and custom persistence strategies. Frameworks shift *where* customization happens:

- **With a low-level API (Anthropic Messages API):** You write the loop, the state machine, the retry logic, the checkpoint format, and the event tracing.
- **With a managed SDK (OpenAI Agents SDK, Claude Agent SDK):** You configure assistants, define tools, and write application-level routing. The SDK handles the inner loop.
- **With a graph runtime (LangGraph):** You define the graph structure, node functions, and state schema. The runtime handles execution and checkpointing.
- **With a multi-agent framework (CrewAI, AG2):** You define agent roles, tasks, and delegation policies. The framework handles message routing.

The "Custom" column reminds you: **frameworks don't eliminate decisions — they shift the abstraction level.** You still choose how to model state, when to interrupt, what to persist, and how to recover from errors. The framework gives you primitives; you build the harness.

<Warning>
**Framework Lock-In**

Choosing a framework is a long-term commitment. Server-side managed services (OpenAI Agents SDK) lock you into provider infrastructure and pricing. Self-hosted frameworks (LangGraph, CrewAI) require operational investment but give you portability. Evaluate not just features but migration cost, vendor dependency, and open-source sustainability before production deployment.
</Warning>

## Trend Lines

Three trends are reshaping agent harnesses:

### Convergence Toward Durable Execution

Early harnesses treated agent loops as ephemeral processes. State lived in memory; crashes lost work. Modern harnesses converge on durable execution:

- **Checkpointing:** LangGraph, Claude Agent SDK, and Temporal all provide checkpoint-based recovery.
- **Exactly-Once Semantics:** Temporal guarantees exactly-once execution for workflow activities.
- **Replay Safety:** Checkpointers enable deterministic replay from any saved state.

This trend is driven by production requirements: agents in customer-facing systems must survive restarts, network failures, and deployment rollouts without losing conversation state or re-executing expensive tool calls.

**Framework examples:**

- **LangGraph** added SQLite and Postgres checkpointers in 2024, enabling production-grade persistence beyond the original in-memory checkpointer.
- **Claude Agent SDK** has file-based checkpointing as a core primitive from its initial release, suited for local or self-hosted deployments.
- **Temporal** has always had durable execution as its foundational design — every workflow step is persisted and replay-safe by default.

### Protocol-Native Tool Access (MCP)

Static tool registration hardcodes tool schemas at runtime startup. MCP (Model Context Protocol) enables dynamic tool discovery:

- **Dynamic Discovery:** Agents query available tools at runtime, adapting to new capabilities without redeployment.
- **Standardized Context:** MCP defines a protocol for exchanging context (e.g., file system access, database schemas) between agents and tools.
- **Cross-Provider Compatibility:** MCP is vendor-neutral, allowing agents to use tools from any MCP-compliant provider.

MCP shifts tools from "baked into the agent" to "discovered from the environment." This reduces coupling and enables runtime adaptability.

**Framework examples:**

- **Claude products** (Claude Desktop, Claude Code) were early MCP adopters, with native MCP support for dynamic tool discovery across desktop and CLI environments.
- **LangGraph** has community-maintained MCP tool adapters that wrap MCP servers as LangChain tools, enabling MCP integration in graph-based agents.
- **OpenAI** announced experimental function-calling-to-MCP bridge capabilities in early 2026, allowing OpenAI function calls to route to MCP servers, though this remains in preview as of mid-2026.

<Callout type="info">
**MCP Adoption**

As of mid-2026, MCP support is growing but not universal. Claude Agent SDK and LangGraph have experimental MCP integrations. OpenAI Agents SDK uses proprietary function calling. Verify current support before assuming MCP availability in your framework.
</Callout>

### Standardized Agent Interoperability (A2A)

Multi-agent systems today use framework-specific message formats. A2A (Agent-to-Agent protocol) aims to standardize:

- **Message Format:** A common schema for agent-to-agent messages, requests, and responses.
- **Handoff Protocol:** Standard semantics for transferring control between agents.
- **Identity and Trust:** Authentication and authorization for agent-to-agent communication.

A2A is early-stage (2026). No production framework has full A2A support yet, but the trend toward standardized multi-agent communication is clear. Future harnesses will interoperate across providers and frameworks using A2A as the coordination layer.

**Framework examples:**

- **Google's A2A proposal** (published in early 2026) defines the initial protocol specification, including message schemas and handoff semantics.
- **LangGraph and CrewAI communities** have early reference implementations of A2A message adapters, demonstrating cross-framework agent communication in experimental settings.
- **No production-grade A2A runtime** exists as of mid-2026 — current implementations are proofs-of-concept without authentication, durable delivery, or error recovery primitives.

## Exercise: Match the Framework

For each scenario below, identify which harness category (Agent SDK, Graph Runtime, Workflow Runtime, or Multi-Agent Runtime) is the best starting point, and name a specific framework.

> **Scenario 1:** You are building a chatbot that helps users navigate a documentation site. The agent needs to search docs, answer questions, and occasionally call a code execution tool. Sessions are short (under 5 minutes) and conversations are independent.

> **Scenario 2:** You are building a content pipeline that receives user requests, assigns research to one agent, writing to another, and editing to a third. Each agent has a specialized system prompt and tools. The pipeline runs for 30+ minutes per request.

> **Scenario 3:** You are building an order fulfillment agent that processes returns. It must check inventory, issue refunds (with human approval for amounts over $200), update the database, and send confirmation emails. If any step fails, prior steps must be rolled back.

> **Scenario 4:** You are building a code review agent that reads a pull request, checks multiple files, runs static analysis, and generates a structured report. The agent's behavior branches based on the type of changes detected (new feature vs. bug fix vs. refactor).

**Suggested answers:**

> **Scenario 1:** Agent SDK — specifically Anthropic Messages API or Claude Agent SDK. Short sessions, no persistence needed, simple tool integration. A graph runtime would be over-engineering.

> **Scenario 2:** Multi-Agent Runtime — specifically CrewAI. Multiple specialized agents with delegation, role-based task assignment, and sequential handoffs. The pipeline benefits from explicit role definitions and task routing.

> **Scenario 3:** Workflow Runtime — specifically Temporal. Durable execution with exactly-once semantics, compensation logic for rollbacks, human approval as a workflow signal, and cross-service orchestration. An agent SDK cannot provide the transactional guarantees this use case demands.

> **Scenario 4:** Graph Runtime — specifically LangGraph. Conditional branching based on state (change type), checkpointing for long runs, and explicit state transitions. The graph structure makes the branching logic visible and debuggable.

## Summary

You now have a map of the agent harness landscape:

- **Four categories:** Agent SDKs / Model-Adjacent Runtimes (Anthropic Messages API, OpenAI Agents SDK, Claude Agent SDK, Vercel AI SDK), Graph Runtimes (LangGraph), Workflow / Orchestration Runtimes (Temporal, Prefect), Multi-Agent Runtimes (CrewAI, AG2, OpenAI handoffs).
- **Comparison table:** Ten concerns across seven harnesses, showing where frameworks provide primitives and where you write custom code.
- **The "Custom" column:** Every harness is custom code; frameworks shift abstraction levels.
- **Ecosystem evolution:** Five phases from raw API calls to protocol-native runtimes.
- **Trend lines:** Durable execution, MCP for dynamic tools, A2A for agent interoperability.
