## Overview

You've built a custom harness from scratch. You traced eight concerns — state, routing, tools, errors, events, interrupts, delegation, and persistence — from first principles through production features. Now you'll see how the industry solves the same problems.

This lesson compares real-world harnesses across the eight-concern framework you learned in Lessons 1–8. You'll categorize frameworks, read a side-by-side comparison table, understand the "Custom" column, track ecosystem evolution, and identify trend lines that matter for production deployments.

<Callout type="info">
**Ecosystem Snapshot**

This comparison reflects the landscape as of mid-2026. Framework capabilities evolve rapidly. Verify against current documentation before production decisions.
</Callout>

## The Big Seven Frameworks

The agent harness ecosystem has converged around seven major frameworks as of mid-2026, each representing a distinct runtime model. Rather than learning each framework in isolation, we compare them across the eight harness concerns you studied in Lessons 1–8.

### 1. Microsoft Agent Framework (MAF)

MAF 1.0 (GA April 2026) is the unified successor to AutoGen and Semantic Kernel. It ships with native MCP and A2A protocol support for both .NET and Python. MAF combines AutoGen's conversational multi-agent abstractions with Semantic Kernel's enterprise features — session-based state management, middleware, telemetry, and type safety — and adds graph-based workflows for explicit control over multi-agent execution paths.

**Runtime model:** Enterprise SDK with graph workflows, middleware pipeline, and protocol-native integrations (MCP + A2A).

### 2. LangGraph

LangGraph models agents as state graphs with explicit nodes (steps), edges (transitions), and typed state schemas. It provides checkpointers for durable execution, interrupt nodes for human-in-the-loop, and time-travel debugging by replaying from any checkpoint.

**Runtime model:** State graph runtime with checkpointers, stores, and visual debugging.

### 3. OpenAI Agents SDK

A managed SDK with a small API surface: you define agents, tools, and handoffs. The SDK manages the inner loop, server-side thread persistence, streaming, and tracing. Despite the name, it supports non-OpenAI model providers.

**Runtime model:** Managed cloud SDK with server-side state and agent handoffs.

### 4. Google Agent Development Kit (ADK)

A code-first toolkit for defining agents, tools, sessions, memory, evaluations, and multi-agent patterns. ADK is protocol-native: it ships with MCP integration for tool discovery and A2A support for cross-framework agent communication. Gemini's multimodal capabilities open use cases that text-only frameworks cannot address.

**Runtime model:** Code-first protocol-native SDK with MCP + A2A and multimodal support.

### 5. Claude Agent SDK / Claude Code Runtime

Anthropic's SDK takes a tool-use-first approach where agents are Claude models equipped with tools, including the ability to invoke other agents as tools. The architecture is deliberately minimal: an agent loop receives a prompt, calls tools as needed (including sub-agent tools), and returns a response. Claude Code extends this with permission systems, context compaction, MCP/plugins/skills/hooks, subagents, and append-oriented session storage.

**Runtime model:** Tool-use-first coding-agent runtime with hierarchical subagent spawning and MCP integration.

### 6. CrewAI

CrewAI models teams of agents with roles, goals, and delegation. You define a crew (e.g., researcher + writer + reviewer), assign tasks, and let agents delegate subtasks to each other. Fastest path to multi-agent prototyping, with pluggable memory/knowledge/RAG backends.

**Runtime model:** Role/team multi-agent framework with task delegation and pluggable backends.

### 7. Mastra

A TypeScript-first agent framework (from the Gatsby team) with built-in workflow orchestration, a dedicated Studio IDE for development and debugging, a Memory Gateway for persistent agent memory, and MCP support. Represents the JS/TS agent-app stack.

**Runtime model:** TypeScript workflow-agent framework with Studio IDE and Memory Gateway.

### Related and Adjacent

These are not among the Big Seven but appear in production agent stacks:

- **AG2 / AutoGen lineage** — the predecessor to MAF's multi-agent abstractions; now in maintenance mode with community forks. Relevant for understanding how multi-agent patterns evolved.
- **Semantic Kernel** — the predecessor to MAF's enterprise features; merged into MAF 1.0. Relevant for .NET-heavy environments still running pre-MAF code.
- **Vercel AI SDK** — React hooks and streaming UI components for integrating LLM agents into frontend applications. Not a harness itself, but a common UI layer atop agent SDKs.
- **Temporal** — a durable execution platform with exactly-once semantics. Hosts agent steps as workflow activities. Use when agents are part of larger distributed workflows requiring transactional guarantees.
- **Prefect / Airflow** — workflow orchestrators for data pipelines. Can embed LLM agent steps but lack interactive session continuity.

## Reading the Comparison Table

The comparison table maps how each framework addresses the eight concerns. Understanding how to read the table helps you evaluate frameworks for your requirements.

**Start with the Runtime Model row.** This row tells you the fundamental shape of each framework: is it a state graph, a managed cloud service, a coding-agent runtime, or a team-based framework? The runtime model determines what kind of customization you do — configuring graph edges vs defining agent roles vs writing loop logic.

**Cell Semantics:** Each cell shows the framework's *approach* to a concern, not a quality rating. "Session-based state with middleware" (MAF) is different from "State graphs with typed schemas" (LangGraph), but neither is inherently better — they reflect different design philosophies and deployment models.

**Managed vs Native vs None:**

- **Native:** The framework handles the concern as a first-class feature. Example: LangGraph provides checkpointers for persistence; MAF provides native A2A for delegation.
- **Supported:** The framework provides hooks or adapters. Example: LangGraph has community MCP adapters; CrewAI supports human-in-the-loop tasks.
- **None:** The framework doesn't support the concern. Example: OpenAI Agents SDK has no native interrupt primitives.

**The "Developer-Defined" Custom Column:** "Developer-defined" means you make all the decisions — which is both the hardest and most flexible option. You choose the state schema, the checkpoint format, the retry policy, the event format, and the persistence backend. Frameworks give you primitives; the Custom column reminds you that even with primitives, you're still writing custom integration code.

**How to Read the Table:** Read row-by-row (by concern) when evaluating frameworks, not column-by-column. For each concern that matters to your application, compare how frameworks handle it. If you need protocol interoperability, read the Protocol Support row. If you need multi-agent coordination, read the Delegation row. If you need durable execution, read the Persistence row. The table is a decision aid, not a scorecard.

## The "Custom" Column

Notice that "Custom" appears in every row. This is not a limitation of the comparison — it's a fundamental truth about agent harnesses.

**Every harness is custom code.** Even when you use a framework, you're writing custom tool definitions, custom routing logic, custom error handlers, custom state schemas, and custom persistence strategies. Frameworks shift *where* customization happens:

- **With a managed cloud SDK (OpenAI Agents SDK):** You configure agents, define tools, and set up handoffs. The SDK handles the inner loop, persistence, and streaming.
- **With an enterprise SDK (MAF 1.0):** You configure middleware, graph workflows, and protocol integrations. The framework handles session state and telemetry.
- **With a graph runtime (LangGraph):** You define the graph structure, node functions, and state schema. The runtime handles execution and checkpointing.
- **With a protocol-native SDK (Google ADK):** You define agents and tools, connect MCP servers and A2A peers. The framework handles discovery and routing.
- **With a coding-agent runtime (Claude Agent SDK):** You define tools and the agent loop. The runtime handles subagent spawning, permissions, and context compaction.
- **With a multi-agent framework (CrewAI):** You define agent roles, tasks, and delegation policies. The framework handles task routing and collaboration.
- **With a TypeScript framework (Mastra):** You define agents, workflows, and memory. The framework handles orchestration and observability.

The "Custom" column reminds you: **frameworks don't eliminate decisions — they shift the abstraction level.** You still choose how to model state, when to interrupt, what to persist, and how to recover from errors. The framework gives you primitives; you build the harness.

<Warning>
**Framework Lock-In**

Choosing a framework is a long-term commitment. Server-side managed services (OpenAI Agents SDK) lock you into provider infrastructure and pricing. Self-hosted frameworks (LangGraph, CrewAI) require operational investment but give you portability. Evaluate not just features but migration cost, vendor dependency, and open-source sustainability before production deployment.
</Warning>

## Trend Lines

Three trends are reshaping agent harnesses:

### Convergence Toward Durable Execution

Early harnesses treated agent loops as ephemeral processes. State lived in memory; crashes lost work. Modern harnesses converge on durable execution:

- **Checkpointing:** LangGraph, Claude Agent SDK, and Temporal all provide checkpoint-based recovery.
- **Exactly-Once Semantics:** Temporal guarantees exactly-once execution for workflow activities.
- **Replay Safety:** Checkpointers enable deterministic replay from any saved state.

This trend is driven by production requirements: agents in customer-facing systems must survive restarts, network failures, and deployment rollouts without losing conversation state or re-executing expensive tool calls.

**Framework examples:**

- **LangGraph** added SQLite and Postgres checkpointers in 2024, enabling production-grade persistence beyond the original in-memory checkpointer.
- **Claude Agent SDK** has file-based checkpointing as a core primitive from its initial release, suited for local or self-hosted deployments.
- **Temporal** has always had durable execution as its foundational design — every workflow step is persisted and replay-safe by default.

### Protocol-Native Tool Access (MCP)

Static tool registration hardcodes tool schemas at runtime startup. MCP (Model Context Protocol) enables dynamic tool discovery:

- **Dynamic Discovery:** Agents query available tools at runtime, adapting to new capabilities without redeployment.
- **Standardized Context:** MCP defines a protocol for exchanging context (e.g., file system access, database schemas) between agents and tools.
- **Cross-Provider Compatibility:** MCP is vendor-neutral, allowing agents to use tools from any MCP-compliant provider.

MCP shifts tools from "baked into the agent" to "discovered from the environment." This reduces coupling and enables runtime adaptability.

**Framework examples:**

- **Claude products** (Claude Desktop, Claude Code) were early MCP adopters, with native MCP support for dynamic tool discovery across desktop and CLI environments.
- **Microsoft Agent Framework 1.0** ships with native MCP support as a first-class runtime primitive — agents can connect to any MCP server for tool discovery without custom adapter code.
- **Google ADK** connects agents to MCP servers alongside A2A, enabling combined tool discovery and agent coordination.
- **LangGraph** has community-maintained MCP tool adapters that wrap MCP servers as LangChain tools.
- **OpenAI** announced experimental function-calling-to-MCP bridge capabilities in early 2026, though this remains in preview.

<Callout type="info">
**MCP Adoption (Mid-2026)**

MCP has reached production adoption. Native MCP support ships in Claude Agent SDK, Microsoft Agent Framework 1.0, and Google ADK. LangGraph has community adapters. OpenAI Agents SDK has an experimental bridge. MCP is production-ready for single-user, local-first environments. Multi-user, server-side, and untrusted-server scenarios require additional security and identity infrastructure.
</Callout>

### Standardized Agent Interoperability (A2A)

Multi-agent systems have historically used framework-specific message formats. A2A (Agent-to-Agent protocol) standardizes cross-runtime agent communication:

- **Message Format:** A common schema for agent-to-agent messages, task delegation, status updates, and results.
- **Handoff Protocol:** Standard semantics for transferring control between agents across different runtimes.
- **Discovery:** Agents discover peers via registries, enabling decentralized multi-agent systems.
- **Identity and Trust:** Authentication and authorization for agent-to-agent communication.

A2A has gained significant traction. The Linux Foundation reported that A2A support grew to over 150 organizations in its first year, including AWS, Cisco, Google, IBM, Microsoft, Salesforce, SAP, and ServiceNow. Production deployments exist in Azure AI Foundry, Amazon Bedrock AgentCore, and Copilot Studio.

**Framework examples:**

- **Microsoft Agent Framework 1.0** ships with native A2A support. Copilot Studio A2A reached general availability, meaning MAF SDK-built agents and Studio-built agents can communicate directly.
- **Google ADK** includes A2A codelabs showing ADK agents exposed as A2A services, with MCP integration alongside A2A for combined tool discovery and agent coordination.
- **LangGraph and CrewAI communities** have early A2A adapters for cross-framework agent communication.
- **Full production A2A** with identity verification, durable delivery, and error recovery is still maturing. Most deployments are within single-vendor ecosystems (Azure, Google Cloud), not truly cross-vendor federations.

## Exercise: Match the Framework

For each scenario below, identify which harness category (Agent SDK, Graph Runtime, Workflow Runtime, or Multi-Agent Runtime) is the best starting point, and name a specific framework.

> **Scenario 1:** You are building a chatbot that helps users navigate a documentation site. The agent needs to search docs, answer questions, and occasionally call a code execution tool. Sessions are short (under 5 minutes) and conversations are independent.

> **Scenario 2:** You are building a content pipeline that receives user requests, assigns research to one agent, writing to another, and editing to a third. Each agent has a specialized system prompt and tools. The pipeline runs for 30+ minutes per request.

> **Scenario 3:** You are building an order fulfillment agent that processes returns. It must check inventory, issue refunds (with human approval for amounts over $200), update the database, and send confirmation emails. If any step fails, prior steps must be rolled back.

> **Scenario 4:** You are building a code review agent that reads a pull request, checks multiple files, runs static analysis, and generates a structured report. The agent's behavior branches based on the type of changes detected (new feature vs. bug fix vs. refactor).

**Suggested answers:**

> **Scenario 1:** Lightweight SDK — specifically Claude Agent SDK or OpenAI Agents SDK. Short sessions, no persistence needed, simple tool integration. A graph runtime or enterprise framework would be over-engineering.

> **Scenario 2:** Multi-agent framework — specifically CrewAI. Multiple specialized agents with delegation, role-based task assignment, and sequential handoffs. The pipeline benefits from explicit role definitions and task routing. MAF 1.0 would also work if you need A2A interoperability or enterprise features.

> **Scenario 3:** Workflow runtime (Temporal) or enterprise SDK (MAF 1.0). Durable execution with exactly-once semantics, compensation logic for rollbacks, human approval as a workflow signal, and cross-service orchestration. A lightweight agent SDK cannot provide the transactional guarantees this use case demands.

> **Scenario 4:** Graph runtime — specifically LangGraph. Conditional branching based on state (change type), checkpointing for long runs, and explicit state transitions. The graph structure makes the branching logic visible and debuggable.

## Summary

You now have a map of the agent harness landscape:

- **The Big Seven:** Microsoft Agent Framework, LangGraph, OpenAI Agents SDK, Google ADK, Claude Agent SDK, CrewAI, Mastra — plus Custom as the fully flexible baseline.
- **Related and adjacent:** AG2/AutoGen lineage, Semantic Kernel, Vercel AI SDK, Temporal, Prefect/Airflow — important but not primary harness frameworks.
- **Comparison table:** Runtime model plus ten concerns across eight harnesses, showing where frameworks provide primitives and where you write custom code.
- **The "Custom" column:** Every harness is custom code; frameworks shift abstraction levels.
- **Ecosystem evolution:** Five phases from raw API calls to protocol-native runtimes.
- **Trend lines:** Durable execution, MCP for dynamic tools (production-adopted), A2A for agent interoperability (150+ organizations, early production).
